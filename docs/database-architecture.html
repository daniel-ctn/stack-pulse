StackPulse — Database & Data Architecture

Status: current  |  Verified against code: 2026-06-04  |  Source of truth: src/db/schema.ts, src/db/index.ts, drizzle/

StackPulse stores GitHub release metadata, AI-generated summaries, user preferences, and auth sessions in PostgreSQL on Neon, accessed through Drizzle ORM and the Neon serverless HTTP driver. This document describes architecture, entity relationships, indexes, data flows, query patterns, optimizations, and operational caveats.

Database client singleton

getDb() lazily constructs a single Drizzle instance wired to Neon HTTP. There is no separate connection pool in application code—the Neon serverless driver issues HTTP requests per query batch. Missing DATABASE_URL throws at first access. 

import { neon } from '@neondatabase/serverless'
import { drizzle } from 'drizzle-orm/neon-http'

db = drizzle(neon(databaseUrl), { schema })

2. High-level architecture

Data moves through four main pipelines:

  1. Auth — GitHub OAuth via Better Auth writes to users, accounts, sessions, verifications.
  2. Ingestion — Cron or custom-repo actions fetch GitHub releases, call OpenRouter for summaries, insert into release_updates, audit in release_fetch_runs.
  3. Personalization — user_tech_preferences scopes the dashboard feed; user_read_releases tracks read state.
  4. Public & marketing — Registry stacks in technologies; public pages aggregate releases; digest_subscribers captures emails (no sender implemented).

3. Entity relationships

Core domain tables and foreign keys (all user-linked rows cascade on user delete):

users ──┬── accounts ├── sessions ├── verifications (Better Auth; no FK to users in schema) ├── user_tech_preferences ──► technologies └── user_read_releases ──► release_updates ──► technologies release_fetch_runs (standalone audit log, no FK) digest_subscribers (standalone; optional stack_slug references technologies.slug logically)

4.6 user_read_releases

Composite PK (user_id, release_id). Insert-on-conflict-do-nothing for bulk mark-read. Unread filter uses LEFT JOIN + IS NULL on read row.

4.7 release_fetch_runs

Audit log per ingestion run: trigger (e.g. cron, custom_repo), status, counts, details jsonb array of per-tech results. No UI yet; table exists for observability.

4.8 digest_subscribers

Email capture for future digest. email unique; optional stack_slug; source default public. Upsert on email updates slug/source. No email cron or sender exists.

5. Indexes & constraints

Explicit indexes are defined in Drizzle schema and applied via migrations. Primary keys and UNIQUE constraints also create btree indexes automatically in PostgreSQL.

5.1 release_updates

NameTypeColumnsPurpose
release_updates_tech_version_unique UNIQUE (tech_id, version) Idempotent inserts; dedupe by tag per stack
release_updates_tech_published_idx BTREE (tech_id, published_at DESC) Per-stack release lists (public stack page)
release_updates_published_idx BTREE (published_at DESC, id DESC) Global feed sort + keyset pagination
release_updates_signals_idx GIN release_signals (jsonb) ? operator for signal filters (breaking, security, etc.)

5.2 Other tables

TableIndexNotes
release_fetch_runs release_fetch_runs_started_idx (started_at DESC) Future admin/history queries by recency
digest_subscribers digest_subscribers_created_idx, digest_subscribers_stack_idx Recency listing; filter by stack_slug
users unique on email Better Auth lookups
technologies unique on name, slug Slug used in public routes and digest validation
user_tech_preferences composite PK only Small cardinality per user; no extra index
user_read_releases composite PK only Join on (user_id, release_id) uses PK

5.3 Indexes not present (gaps to know)

  • No dedicated index on technologies.slug beyond UNIQUE (sufficient for equality).
  • No full-text search (tsvector) index — feed search uses ILIKE across multiple columns and jsonb cast to text.
  • No index on importance_level alone; combined with tech_id / date filters in practice.
  • Better Auth tables rely on PK; session token lookups depend on Better Auth query patterns.

6. Enums & JSONB conventions

importance_level PostgreSQL enum: low, medium, high, critical.

JSONB array columns store structured AI output. release_signals holds normalized tags; feed UI also derives signals from non-empty jsonb arrays (breaking_changes, etc.) for backward compatibility.

Signal filter SQL uses GIN-friendly release_signals ? 'breaking' plus fallback jsonb_array_length checks.

8. Data flows

8.1 Authentication

  1. User signs in with GitHub → Better Auth API route.
  2. drizzleAdapter(getDb()) maps schema: user, account, session, verification tables.
  3. Session cookie validated on server components and API routes via getAuth().api.getSession().
  4. Account deletion: delete(users) cascades preferences, read state, sessions, accounts.

8.2 Release ingestion (write path)

  1. Cron (GET /api/cron/fetch-releases): Bearer CRON_SECRET; loads distinct technologies joined to user_tech_preferences (only followed stacks).
  2. Creates release_fetch_runs row; processes techs in parallel chunks of 6 (CHUNK_SIZE) to balance API limits and Vercel maxDuration = 60.
  3. Per tech (processTechReleases): fetch up to 5 GitHub releases; skip drafts / missing published_at.
  4. Check existing row by (tech_id, version); if absent, call OpenRouter summarizeRelease, then insert with onConflictDoNothing on unique constraint.
  5. Custom repo: addCustomTech upserts technology, links preference, immediately runs same ingestion with trigger custom_repo.
  6. Run finishes with aggregated stats in release_fetch_runs.details.

8.3 Dashboard feed (read path)

  1. SSR: dashboard/page.tsx calls getReleaseFeedPage + tech options.
  2. Client: TanStack Query hits GET /api/releases with same filters and cursor for infinite scroll.
  3. Scope: authenticated users filter tech_id IN (followed ids); anonymous users see all releases (public scope).
  4. Pagination: keyset cursor (base64url JSON of publishedAt + id), page size 30, ORDER BY published_at DESC, id DESC.
  5. Unread: LEFT JOIN user_read_releases with sentinel user id for public scope.

8.4 Read / unread

markReleasesRead verifies releases belong to user's followed techs before insert. markReleaseUnread deletes the junction row.

8.5 Public stacks (SEO)

  • getPublicStackPage: lookup by slug, then releases ordered by published_at (uses tech_published index).
  • getPublicStackIndex: GROUP BY all technologies with aggregate counts and signal filters — can be heavy as data grows.

8.6 Release advice (AI read)

POST /api/release-advice loads one release joined to preferences (must follow stack). Rate limited in memory per user. Uses stored summary + raw body for context.

8.7 Digest signup (write-only queue)

Server action upserts digest_subscribers. In-memory IP rate limit (not durable across serverless instances).

9. Query patterns & performance

9.1 Feed query (getReleaseFeedPage)

  • Joins: release_updates INNER JOIN technologies; optional LEFT JOIN read table.
  • Filters: importance IN list, signal (GIN/jsonb), tech slug, ILIKE search, keyset cursor.
  • Fetches limit + 1 rows to detect next page without COUNT(*).
  • Index alignment: sort matches release_updates_published_idx; user scope adds tech_id filter (tech_published index helps when filtering one stack).

9.2 Ingestion deduplication

Pre-insert SELECT by tech_id + version, then insert with conflict target — redundant but avoids wasted AI calls when row exists. Race: two workers could both pass SELECT; unique constraint + onConflictDoNothing prevents duplicates.

9.3 Neon HTTP driver implications

  • Each getDb() query round-trip is HTTP—batch related reads in one handler where possible (dashboard uses Promise.all).
  • Long cron jobs issue many sequential queries per release (select + insert); dominated by GitHub + AI latency, not DB.
  • No transactions wrap multi-step preference saves (delete then insert loops)—partial failure could leave inconsistent prefs until retry.

9.4 Search performance caveat

Text search uses leading-wildcard ILIKE '%term%' on several text columns and ::text on jsonb. This cannot use standard btree indexes efficiently. Acceptable at small/medium scale; consider PostgreSQL full-text search or trigram indexes if search becomes hot.

9.5 Storage growth

raw_release_body stores full release notes per row. Cron fetches 5 releases per tech per run but only inserts new versions—table grows with unique tags over time.

10. Technical caveats

Serverless rate limits (in-memory) Digest signup and release-advice rate limits use process-local Maps. On Vercel, limits are best-effort per instance, not global.
Cron scope Only technologies with at least one follower are fetched. Registry stacks with zero followers never get releases until someone follows them.
Custom technology slug collision Custom repos derive slug from display name. Two users naming repos similarly can share one technologies row (by design when slug matches).
Public feed when signed out /api/releases without session uses scope: public — all releases, not personalized. Read filter forced to all.
AI before insert Summarization runs before insert; failed AI skips row (errors counted per tech). Existing rows are never re-summarized on cron unless backfill script is run.
Null published_at Ingestion skips releases without published_at. Feed ordering treats cursor pagination with timestamps; null dates sort unpredictably—ingestion avoids nulls for new rows.
No row-level security Authorization is application-layer (session checks, preference joins). Direct DB access bypasses app rules.
Digest & fetch run UI Tables exist; weekly digest email pipeline and fetch-run history UI are not implemented.
