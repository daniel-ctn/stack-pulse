3. Dashboard feed architecture

dashboard/page.tsx (RSC) ├─ getSession() ├─ parse searchParams (same parsers as API) ├─ getReleaseFeedPage() → initialPage └─ getUserTechOptions() / getAllTechOptions() │ ▼ ReleaseFeed ('use client') ├─ nuqs: importance, read, signal, tech, q ├─ TanStack Query: GET /api/releases?cursor=... ├─ useVirtualizer: long lists └─ server actions: markReleasesRead, markReleaseUnread

Hydration contract

  • Server passes initialPage matching current URL filters.
  • Client query uses same filter key in queryKey: ['release-feed', ...].
  • Filter changes reset cursor and refetch from API.
  • Parsers live in release-feed-types.ts — must stay in sync with nuqs parsers on client.

6. Loading & error UX

  • Route-level loading.tsx skeletons under dashboard (frame + skeleton pattern).
  • PulseLoader for inline loading — not generic spinners.
  • ErrorView for error boundaries.
  • prefers-reduced-motion respected in global CSS.

7. Public stacks pages

Fully server-rendered — no TanStack Query. Data from public-stacks.ts. JSON-LD in page for SEO. Signal badges derived server-side to match feed semantics.

8. Auth UI flow

Sign-in page triggers Better Auth client redirect. Post-auth: middleware-less flow — pages check session and redirect empty stacks to /onboarding.

9. Extension checklist (structure)

  • New filter? Update release-feed-types.ts, release-feed.ts, nuqs client, and API route.
  • New mutation? Prefer server action in actions.ts with { ok, error } pattern.
  • New paginated list? Reuse keyset cursor pattern, not offset.
  • Keep product icons on hugeicons-react per design system.
