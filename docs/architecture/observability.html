3. Using release_fetch_runs for ops

Each cron or custom-repo ingestion creates a row:

ColumnMeaning
triggercron or custom_repo
statusrunning → completed or completed_with_errors
technologies_scannedCount of techs processed
releases_insertedSum of new rows
errorsPer-tech error count aggregate
detailsJSON array: { tech, inserted, errors } per stack
started_at / finished_atDuration analysis
 
-- Example: recent cron runs (run in SQL client / Studio)
SELECT id, trigger, status, releases_inserted, errors, started_at, finished_at
FROM release_fetch_runs
ORDER BY started_at DESC
LIMIT 20;

4. Manual verification playbook

  1. pnpm exec tsc --noEmit && pnpm lint && pnpm build
  2. Trigger cron locally: curl -H "Authorization: Bearer %CRON_SECRET%" http://localhost:3000/api/cron/fetch-releases
  3. Inspect JSON response results array for per-tech errors
  4. Confirm new rows in release_updates for expected tags

See docs/operations/verification.md for full checklist.

6. Recommended future observability

  • Admin page for release_fetch_runs (last N runs, error highlight).
  • Sentry (or similar) on API routes + server actions with release/tag context.
  • Structured logs with runId, techId on ingestion paths.
  • Cron alert — Vercel integration or external ping on non-200 cron response.
  • OpenRouter usage — track token/cost per run if billing becomes a concern.
