Security architecture

Status: current · Last verified: 2026-06-04

No Postgres RLS Authorization is enforced in application code (session checks, preference joins). Anyone with direct DB access bypasses app rules.

Never commit .env. Generate secrets with openssl rand -hex 32 per .env.example.

3. Data isolation

  • Feed (authenticated): inArray(tech_id, followed ids) from user_tech_preferences.
  • Mark read: Inner join ensures release belongs to a followed tech before insert into user_read_releases.
  • Release advice: Same preference join on target release.
  • Public API scope: Unauthenticated /api/releases returns global release set — intentional for browsing, not a leak of user-specific data.

5. Cron hardening

  • Route returns 500 if CRON_SECRET unset.
  • 401 if bearer missing or wrong.
  • timingSafeEqual on secret comparison (length check first).

7. AI / prompt injection

  • Release bodies from GitHub are untrusted content passed as model input.
  • Structured outputs validated with Zod before DB insert or JSON response.
  • Advice endpoint does not execute code from model output.

8. Account deletion

deleteAccountAction deletes users row; FK cascades remove sessions, accounts, preferences, read state. Releases and technologies remain (shared registry/custom rows).

9. Privacy & SEO boundaries

  • Dashboard route group: robots: noindex on layout.
  • Public stack pages indexable for SEO.
  • Digest table stores email addresses — treat as PII at rest (Neon encryption at rest depends on provider).
